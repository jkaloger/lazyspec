## Acceptance Criteria

### AC: incremental-default-numbering

Given a type definition with no `numbering` field in `.lazyspec.toml`
When `lazyspec create` is run for that type
Then the filename uses zero-padded incremental numbering starting from 001

### AC: incremental-scans-existing-files

Given a directory containing `RFC-003-foo.md` and `RFC-005-bar.md`
When `next_number` is called with prefix `RFC`
Then it returns 6, one greater than the highest existing number

### AC: sqids-id-generation

Given a type configured with `numbering = "sqids"` and a valid `[numbering.sqids]` section
When `lazyspec create` is run for that type
Then the filename contains a sqids-encoded alphanumeric ID instead of a zero-padded number

### AC: sqids-output-is-lowercase

Given sqids numbering is configured
When a sqids ID is generated by `next_sqids_id`
Then the ID string is entirely lowercase

### AC: sqids-salt-changes-alphabet

Given two different `salt` values in `[numbering.sqids]`
When `shuffle_alphabet` is called with each salt
Then the resulting alphabets differ, producing different IDs from the same numeric input

### AC: sqids-collision-retry

Given a directory containing a file whose name matches the first candidate sqids ID
When `next_sqids_id` is called
Then the input is incremented and a different, non-colliding ID is returned

### AC: sqids-min-length-enforced

Given a `[numbering.sqids]` section with `min_length = 6`
When a sqids ID is generated
Then the ID is at least 6 characters long

### AC: resolve-filename-no-number-placeholder

Given a naming pattern that contains no `{n:03}` or `{n}` placeholder
When `resolve_filename` is called
Then the filename is returned without consulting any numbering strategy

### AC: resolve-filename-pre-computed-id

Given a `pre_computed_id` is provided to `resolve_filename`
When the pattern contains a number placeholder
Then the pre-computed ID is substituted directly without scanning the directory or generating a new ID

### AC: reserved-numbering-atomic-push

Given a project configured with `numbering = "reserved"` and a reachable remote
When `reserve_next` is called
Then it queries `refs/reservations/{PREFIX}/*` on the remote, creates a local ref, and atomically pushes it before returning the reserved number

### AC: reserved-push-rejection-retry

Given a concurrent reservation where the atomic push is rejected
When the push fails with a rejection or already-exists error
Then the local ref is cleaned up, the candidate is incremented, and the push is retried up to `max_retries` times

### AC: reserved-retries-exhausted

Given the retry loop has exhausted all `max_retries` attempts
When every push is rejected
Then `reserve_next` returns an error naming the prefix and the range of attempted numbers

### AC: reserved-remote-unreachable

Given the remote is unreachable (offline, auth failure, DNS failure)
When `ls_remote` is called for reservation queries
Then it fails with an error indicating the remote is unreachable and suggests `--numbering incremental` or `--numbering sqids` as overrides

### AC: list-reservations-returns-all-refs

Given reservation refs exist on the remote under `refs/reservations/*`
When `list_reservations` is called
Then it returns a `Reservation` for each ref containing the parsed prefix, number, and ref path

### AC: delete-remote-ref-removes-reservation

Given a reservation ref path exists on the remote
When `delete_remote_ref` is called with that ref path
Then the ref is removed from the remote via `git push --delete`
